Technique

Cette section présente les compétences techniques développées lors du stage, illustrées par des traces concrètes et évaluées dans un bilan réflexif.

Trace 01

Architecture du pipeline de données

Trace 1 : Schéma d'architecture du pipeline de données initial (RDS → S3 → RDS → Redshift → QuickSight)

Contexte

Au début du stage, j'ai réfléchis à une architecture pour automatiser la remontée des erreurs depuis la base de données de production vers un dashboard QuickSight. Cette architecture initiale prévoyait un pipeline en 5 étapes utilisant plusieurs services AWS.

Savoir-faire mobilisés

01
Analyse d'architecture cloud
Étude de l'environnement technique existant et compréhension des interactions entre les différents services AWS pour répondre au besoin d'automatisation.
02
Comparaison de solutions techniques
Évaluation de deux approches : logiciel complet vs pipeline de données automatisé. Choix de la solution la plus adaptée aux contraintes de temps (8 semaines) et de maintenance.
03
Schématisation de flux de données
Formalisation précise de chaque étape du flux, des formats de données et des transformations nécessaires à l'aide d'Excalidraw.
04
Choix d'architecture AWS
Sélection de chaque service AWS en fonction de son rôle spécifique : RDS pour la base relationnelle, S3 pour le stockage intermédiaire, Redshift pour l'analytique columnar, QuickSight pour la visualisation.

Cette architecture initiale a été abandonnée en semaine 2 au profit d'une approche 0-ETL plus simple, mais elle m'a permis de comprendre en profondeur les services AWS et les contraintes du projet.

Trace 02

Vues matérialisées SQL

Trace 2 : Extrait du code SQL d'une vue matérialisée déployée sur Redshift

Contexte

Après le pivot vers l'approche 0-ETL, j'ai transformé les 17 requêtes SQL de détection d'erreurs en vues matérialisées directement sur Redshift. Ces vues sont déployées via le CDK de l'entreprise et se rafraîchissent automatiquement.

Savoir-faire mobilisés

01
Écriture de requêtes SQL complexes
Transformation de requêtes de détection d'anomalies en vues matérialisées optimisées, avec gestion des jointures, agrégations et filtres temporels.
02
Optimisation de performances SQL
Identification et résolution des goulots d'étranglement : indexation, partitionnement des données, optimisation des jointures pour réduire les temps d'exécution.
03
Respect de standards de qualité
Adaptation du code aux standards de l'entreprise : formatage, nommage, commentaires, passage des tests qualité automatiques avant déploiement.

En semaine 8, j'ai également créé une vue complète de zéro sans modèle de départ, démontrant ma maîtrise de la structure des tables et de la logique métier.

Trace 03

Déploiement CDK et Git

Trace 3 : Pull Request sur le dépôt Git de l'entreprise avec tests qualité et déploiement CDK

Contexte

Le déploiement des vues matérialisées et des DataSets QuickSight s'effectue via le CDK (Cloud Development Kit) d'AWS, avec un workflow Git professionnel incluant des tests qualité automatiques.

Savoir-faire mobilisés

01
Utilisation de Git en entreprise
Maîtrise du workflow professionnel : branches dédiées, commits atomiques, Pull Requests, code review, gestion des conflits de merge.
02
Déploiement via CDK
Ajout de nouveaux DataSets au CDK existant, compréhension de l'infrastructure as code, déploiement automatisé sur les environnements AWS.
03
Debug de pipeline CI/CD
Résolution des erreurs de tests qualité automatiques : parseurs SQL, vérification des types, formatage du code, interprétation des logs de déploiement.

La mise à jour de la base de test en semaine 6 a rendu ma branche obsolète, nécessitant deux jours de résolution de conflits avec l'aide d'un collègue. Cette expérience m'a appris l'importance de la communication et de la collaboration inter-équipes.
