Savoir-faire 01

Concevoir et déployer un pipeline de données cloud

Auto-évaluation

Bon

Synthèse

J'ai conçu une architecture complète de pipeline de données utilisant les services Amazon Web Services (AWS) (RDS, S3, Redshift, QuickSight). Suite au pivot vers le 0-ETL (sans ETL — Extract, Transform, Load), j'ai adapté cette architecture pour utiliser des vues matérialisées directement sur Redshift, simplifiant considérablement la solution. J'ai aussi utilisé Docker/MinIO pour simuler S3 en local avant d'obtenir les accès AWS, ce qui m'a permis d'avancer rapidement durant la première semaine.

Ce savoir-faire repose sur quatre savoir-faire élémentaires illustrés par la Trace 1 : analyser un environnement technique, comparer des solutions, schématiser un flux de données, et simuler un environnement cloud en local.

Contexte d'apprentissage

L'architecture cloud AWS était entièrement nouvelle pour moi. Les cours de base de données à l'IUT m'avaient donné des bases en SQL et en modélisation (MCD — Modèle conceptuel de données), mais l'aspect cloud et les services managés étaient inconnus. J'ai appris principalement lors du stage, en étudiant la documentation AWS, en faisant des essais-erreurs, et en échangeant avec mes collègues. L'IA (intelligence artificielle) m'a aussi aidé à comprendre certains concepts et à écrire le script Python de 1600 lignes, utilisé pour l'achitecture pensée lors de la première semaine.

Difficultés rencontrées

La principale difficulté a été de comprendre les contraintes de chaque service AWS (coût, performance, maintenance) et de choisir la bonne architecture sans avoir accès à la vraie base de données. Le pivot vers le 0-ETL en semaine 2 a été frustrant car j'avais passé une semaine complète à développer un script Python qui n'était plus utile, mais cette expérience m'a appris à chercher la solution la plus simple plutôt que de commencer sans avoir suffisamment réfléchi à l'architecture.

Utilisation de l'IA

J'ai utilisé l'IA (OpenCode Go avec Kimi 2.6 et GLM) pour m'aider à écrire le script Python. L'IA ne m'a pas remplacé : je comprenais bien la logique. Elle m'a surtout fait gagner du temps sur la syntaxe, le débuggage et m'a suggéré des approches que je ne connaissais pas (comme MinIO pour simuler S3). C'était un assistant.

Auto-évaluation nuancée

Sur ce savoir-faire, je me sens aujourd'hui capable de concevoir une architecture de données cloud et de choisir les services adaptés. La Trace 1 en est l'exemple, et le pivot vers le 0-ETL montre que j'ai aussi su remettre en question mon architecture initiale pour une solution plus simple.

Je tiens toutefois à cadrer cette compétence. Elle est convenable dans le périmètre des services AWS que j'ai utilisés (RDS, S3, Redshift, QuickSight), mais en un peu plus de deux mois, je n'ai pas pu me familiariser parfaitement avec tout ce système que je venais de découvrir. De plus, face à un projet cloud utilisant d'autres providers ou des services que je ne maîtrise pas encore, l'adaptation serait plus difficile. Si je devais refaire ce stage, j'essaierais de moins foncer tête baissée et de m'assurer que l'architecture est la bonne dès le début, plutôt que de développer pendant une semaine avant de pivoter.

Avant le stage

Connaissances basiques en bases de données relationnelles (SQL, MCD) acquises en BUT 1 et BUT 2. Aucune expérience avec les services cloud, Docker, ou les architectures distribuées. Python avancé maîtrisé, mais pas appliqué à des projets de données.

Après le stage

Capacité à concevoir une architecture de données cloud complète, à choisir les services adaptés, et à déployer via des outils d'infrastructure as code (CDK). Bonne maîtrise de Redshift et de l'approche 0-ETL. Découverte de Docker et MinIO. Compréhension de l'importance de la simplicité dans les architectures.

Avant le stage : débutant → Après le stage : autonome (dans le périmètre AWS)

Savoir-faire 02

Écrire et optimiser des requêtes SQL en production

Auto-évaluation

Bon

Synthèse

J'ai transformé les requêtes de détection d'erreurs en vues matérialisées sur Redshift, en respectant les contraintes de performance et les standards de qualité de l'entreprise. J'ai aussi créé une vue complète de zéro en semaine 8 (vue des soldes de comptes bénéficiaires), sans modèle de départ. J'ai dû gérer des problèmes syntaxe spécifique à RedShift, des fonctions mutables non acceptées par le parseur, et une logique métier complexe (reversals, mouvements de fonds).

Ce savoir-faire repose sur trois savoir-faire élémentaires illustrés par la Trace 2 : écrire / comprendre des requêtes complexes, transformer en vues matérialisées, et respecter des standards de qualité stricts.

Contexte d'apprentissage

Avant le stage, j'avais un niveau SQL basique (SELECT, JOIN, GROUP BY) acquis en BUT 1 et BUT 2. Les vues matérialisées et les spécificités de Redshift étaient entièrement nouvelles. C'était la première fois que j'avais des vues à créer, j'ai donc eu recours à l'IA pour comprendre la syntaxe, mais aussi aux autres vues existantes de l'entreprise pour m'aider à transformer ces requêtes au début. J'ai appris beaucoup en corrigeant les erreurs des tests qualité et en lisant le code existant.

Difficultés rencontrées

Les tests de qualité automatiques étaient très stricts et généraient des erreurs difficiles à interpréter. J'ai passé beaucoup de temps à fixer des bugs de déploiement liés à des fonctions mutables comme CURRENT_DATE qui n'étaient pas acceptées par le parseur de Redshift. La compréhension de la logique métier derrière chaque requête a aussi été un défi, notamment pour les reversals (annulations de transactions) et les mouvements de fonds (transferts entre comptes internes).

En semaine 2, j'ai aussi découvert que les requêtes étaient trop lourdes pour la base de production : mon script Python utilisait des fetchall() qui accumulaient toutes les données en mémoire et faisaient planter le processus. J'ai dû repenser la gestion de la mémoire et simplifier la structure de 63 à 11 colonnes.

Utilisation de l'IA

J'ai utilisé l'IA un peu pour cette partie, principalement au début quand je ne savais pas comment créer des vues matérialisées. L'IA m'a aidé à comprendre la syntaxe et les bonnes pratiques, mais je me suis aussi beaucoup appuyé sur les vues existantes de l'entreprise pour apprendre par l'exemple. L'IA était un complément, pas une béquille.

Auto-évaluation nuancée

Sur ce savoir-faire, je me sens aujourd'hui autonome : je sais écrire des requêtes SQL complexes pour Redshift, les transformer en vues matérialisées, et les faire passer les tests de qualité automatiques. La création d'une vue complète de zéro en semaine 8 montre que j'ai bien intégré la structure des tables et la logique métier.

Ce que j'en retiens surtout, c'est l'importance de comprendre la logique métier avant de coder. Les requêtes impliquent des concepts complexes qu'il fallait maîtriser pour écrire du code correct. J'ai aussi appris à anticiper les contraintes de performance dès la conception, et à ne pas sous-estimer la différence entre une base de test et une base de production volumineuse.

Je pensais être plus à l'aise avec SQL avant le stage, mais la réalité de la production m'a montré que j'avais encore beaucoup à apprendre. Par contre, je suis assez fier de ma progression et de ma capacité à créer une vue complexe de zéro en fin de stage.

Avant le stage

Maîtrise du SQL de base (SELECT, JOIN, GROUP BY) acquise en BUT 1 et BUT 2. Pas d'expérience avec les bases de données de production volumineuses, les vues matérialisées, ou les spécificités de Redshift.

Après le stage

Capacité à écrire des requêtes SQL complexes pour Redshift, à gérer les problèmes de mémoire sur des bases volumineuses, et à créer des vues matérialisées respectant des standards de qualité stricts. Bonne compréhension de la logique métier du domaine bancaire/transport (reversals, mouvements de fonds, soldes).

Avant le stage : débutant → Après le stage : autonome

Savoir-faire 03

Versionner et déployer en environnement professionnel

Auto-évaluation

Moyen

Synthèse

J'ai travaillé avec le workflow Git professionnel de Kuba : branches dédiées (feat/reporting-cch puis feat/cch-checking après un stash), Pull Requests, tests qualité automatiques (syntaxe et parseurs), et déploiement via CDK. J'ai ajouté des DataSets au CDK existant et débuggé des erreurs de déploiement. La mise à jour de la base de test en semaine 6 a rendu le déploiement très compliqué, nécessitant l'aide de mon collègue Aurélien Bak pour modifier le CDK et comprendre les erreurs.

Ce savoir-faire repose sur quatre savoir-faire élémentaires illustrés par la Trace 3 : utiliser Git en entreprise, déployer via CDK, débugger des erreurs CI/CD, et gérer des conflits de branche.

Contexte d'apprentissage

Avant le stage, j'utilisais déjà Git avec des branches pour les projets de l'IUT, mais je n'avais jamais travaillé avec des PR (Pull Requests), de la code review, ou du CI/CD. Le workflow professionnel était donc nouveau pour moi. J'ai appris sur le terrain, en suivant les méthodes de l'entreprise et en demandant de l'aide à mes collègues. Mon collègue Aurélien Bak, développeur depuis longtemps dans la boîte qui a mis au point les CDK GitHub et a déjà travaillé des vues comme les miennes, m'a beaucoup aidé à comprendre le processus.

Difficultés rencontrées

Le CDK était trop complexe pour mon niveau actuel. Quand je n'arrivais pas à comprendre les erreurs de déploiement, j'envoyais parfois les logs à l'IA pour rester autonome, mais ses réponses étaient souvent approximatives. J'ai donc appris à ne pas trop me fier à l'IA pour ce type de problèmes complexes et à demander de l'aide à mes collègues qui connaissaient mieux l'infrastructure de l'entreprise.

En semaine 6, une mise à jour de la base de test a rendu le déploiement très compliqué. Plusieurs collègues y ont passé du temps et nous sommes restés bloqués dessus. C'est un peu flou pour moi car je n'étais pas un acteur majeur de cette résolution, mais cela a cassé le CDK et j'ai dû solliciter Aurélien Bak pour m'aider à le modifier et comprendre les erreurs. Cette expérience m'a montré que le CDK était complexe et que je devais solliciter de l'aide sur les parties les plus techniques.

Utilisation de l'IA

J'ai utilisé l'IA pour cette partie, principalement quand je n'arrivais pas à comprendre les erreurs de déploiement. Je lui envoyais les logs pour essayer de rester autonome, mais ses réponses étaient souvent approximatives. J'ai donc appris à ne pas trop me fier à l'IA pour ce type de problèmes complexes et à demander de l'aide à mes collègues qui connaissaient mieux l'infrastructure de l'entreprise.

Auto-évaluation nuancée

Sur ce savoir-faire, je me sens aujourd'hui à l'aise avec le workflow Git professionnel (branches, PR, code review) et je comprends le processus de déploiement via CDK. Par contre, je ne suis pas encore autonome sur les parties les plus complexes du CDK, et j'ai besoin d'aide pour résoudre certains conflits ou erreurs de déploiement.

Ce que j'en retiens, c'est la différence entre le Git scolaire et le Git professionnel. Les tests qualité automatiques, les PR obligatoires et le déploiement continu m'ont appris à écrire du code lisible, documenté, et à justifier mes choix techniques. Je suis assez fier de ma maîtrise de Git et des branches, j'ai pu très rapidement utiliser ces compétences. Par contre, le CDK reste un domaine où j'ai encore beaucoup à apprendre.

Cette expérience m'a aussi appris l'importance de la communication et de la collaboration inter-équipes. Certains problèmes ne peuvent être résolus seul, et savoir à qui s'adresser est une compétence en soi.

Avant le stage

Utilisation de Git avec des branches pour les projets de l'IUT. Pas d'expérience avec les PR, la code review, le CI/CD, ou l'infrastructure as code (CDK).

Après le stage

Maîtrise du workflow Git professionnel (branches, PR, code review). Compréhension du processus de déploiement via CDK, même si je ne suis pas encore autonome sur les parties les plus complexes. Bonne habitude des tests qualité automatiques. Capacité à demander de l'aide de façon ciblée.

Avant le stage : intermédiaire (Git) → Après le stage : moyen (Git + CDK)
